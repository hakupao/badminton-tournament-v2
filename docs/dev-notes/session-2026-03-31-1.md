# 开发日志 2026-03-31 Session 1 — D1 archived 状态迁移修复

## 背景

本次变更用于修复赛事状态新增 `archived` 后，Cloudflare D1 远程数据库无法安全执行迁移的问题。

触发场景如下：

- 代码层已经支持 `tournaments.status = 'archived'`
- 远程 D1 实际表结构仍停留在旧的 `CHECK(status IN ('draft', 'active', 'finished'))`
- 管理后台执行归档操作时，远程库会因为约束未更新而返回 500

---

## 问题定位

本次排查确认了两个独立问题：

### 1. 旧迁移执行方式不适用于 D1 远程环境

- 逐条在 Dashboard SQL Console 手动执行迁移时，`DROP TABLE tournaments` 会触发外键约束错误
- 根因是多个表仍然通过外键引用 `tournaments`，而 `PRAGMA` 状态不会跨“逐条手动执行”持续生效

受影响的引用表包括：

- `groups`
- `players`
- `template_positions`
- `template_matches`
- `matches`
- `tournament_participants`

### 2. 远程 D1 不接受 SQL 中显式事务

- `wrangler d1 execute --remote --file=...` 在执行包含 `BEGIN TRANSACTION` / `COMMIT` 的 SQL 文件时失败
- 远程返回明确错误：D1 后端已自行管理事务边界，不允许在 SQL 文件中显式开启事务

---

## 解决方案

### 1. 重写迁移脚本执行策略

文件：

- `scripts/migrate-add-archived-status.sql`

关键调整：

- 移除 `BEGIN TRANSACTION` / `COMMIT`
- 保留 `PRAGMA defer_foreign_keys = on`
- 新增 `DROP TABLE IF EXISTS tournaments_new`，清理失败迁移留下的中间表
- 将 `INSERT INTO tournaments_new SELECT * FROM tournaments` 改为显式列清单，避免未来字段顺序变化导致拷贝风险
- 补充脚本注释，明确要求“整文件一次执行”，禁止在 Dashboard 中逐句运行

### 2. 固化远程迁移命令

文件：

- `package.json`

新增脚本：

- `d1:migrate:archived`

作用：

- 统一通过 `wrangler d1 execute shuttle-arena-db --remote --file=scripts/migrate-add-archived-status.sql` 执行远程迁移
- 避免使用 `--command \"$(cat ...)\"` 时被 SQL 注释干扰参数解析

## 运维说明

后续如果需要再次执行本次迁移，使用以下命令：

```bash
npm run d1:migrate:archived
```

注意事项：

- 不要在 Cloudflare Dashboard 的 SQL Console 中逐条粘贴执行该迁移
- 不要重新引入 SQL 层面的 `BEGIN TRANSACTION` / `COMMIT`
- 若迁移中断，脚本会优先清理 `tournaments_new` 中间表后再继续
