# ShuttleArena 项目环境适配说明

## 这份文档的用途

这份文档用于说明本项目在 **Windows / macOS 本地开发** 与 **GitHub -> Cloudflare Pages 自动部署** 场景下的兼容性约束、已做过的适配工作、以及后续修改时必须遵守的边界。

以后无论是和 AI 协作，还是和新的开发者协作，都可以直接引用这份文档，避免重复解释项目背景。

---

## 项目背景

本项目在以下环境中被持续开发和调整过：

- macOS Apple Silicon（M 系列芯片）
- macOS Intel 芯片
- 不同版本的 macOS
- Windows 11
- GitHub 推送后由 Cloudflare Pages 在 Linux 环境自动构建和部署

另外，本项目曾接受过多个 AI agent 的辅助编程，因此代码中历史上可能出现过：

- 死代码
- 无用代码
- 冗余代码
- 半成品思路残留
- 废弃实现
- 单文件过于臃肿的情况

项目已经做过一轮“保守瘦身”和兼容性整理，但后续继续优化时，仍然必须遵守下面的原则。

---

## 不可破坏的前提

后续任何清理、重构、优化，都必须满足以下要求：

1. **不能改变现有功能表现**
2. **不能改变既有项目逻辑**
3. **必须保证 macOS 和 Windows 本地都能继续开发和运行**
4. **必须保证推送到 GitHub 后，Cloudflare Pages 能继续自动构建和部署**
5. **环境变量和 Cloudflare D1 绑定按现有配置继续工作**

也就是说，这个项目允许做：

- 删除未使用代码
- 删除未引用资源
- 收紧类型
- 拆分臃肿文件
- 清理冗余依赖
- 提高健壮性

但不允许做：

- 擅自改功能
- 擅自改数据流
- 擅自改部署链路
- 擅自替换关键运行时方案
- 只在单一平台验证就提交

## 当前采用的兼容性方案

### 1. 本地开发主路径

本项目的本地开发主路径是：

- `npm run dev`

它会先执行：

- `npm run d1:init:local`
- `npm run d1:seed:local`

然后以 Cloudflare 本地绑定模式启动 Next.js 开发环境。

这意味着日常本地联调优先依赖：

- 本地 D1
- Wrangler 本地绑定
- `USE_D1=true`

因此，后续修改时不要轻易破坏：

- `package.json` 中的 `predev` / `dev` / `dev:fast`
- `wrangler.toml`
- 本地 D1 初始化与 seed 流程

### 2. 本地 Node fallback 仍然保留

项目中仍保留了 `better-sqlite3` 的本地 fallback，用于非 Edge Node.js 场景的兜底。

当前对应逻辑位于：

- [src/db/index.ts](/Users/bojiangzhang/MyProject/badminton-tournament-v2/src/db/index.ts) — App 运行时使用的 Edge-safe D1 入口
- [src/db/node.ts](/Users/bojiangzhang/MyProject/badminton-tournament-v2/src/db/node.ts) — Node-only 的 `better-sqlite3` fallback

注意事项：

- 不能随意删除 `better-sqlite3`
- 不能把数据库能力改成只支持单一运行时
- 不能让 Edge route 静态 import 到 Node fallback
- 不能把 Cloudflare D1 和 Node fallback 混成一套不可控逻辑

当前已经把早期的 `eval('require(...)')` 改成了 **Node builtin module loader** 方案，以减少 Cloudflare bundling 警告。

### 3. Cloudflare Pages 构建链路

当前 Cloudflare 构建链路依赖：

- `@cloudflare/next-on-pages`
- `.vercel/output/static`
- `wrangler.toml` 中的 `pages_build_output_dir = ".vercel/output/static"`

当前构建命令：

- `npm run build:cf`

部署命令：

- `npm run deploy`

后续如果要动这条链路，必须先完整验证：

- `npm run lint`
- `npm run build`
- `npm run build:cf`

### 4. `middleware.ts` 目前是有意保留的

虽然 Next.js 16 会提示：

- `The "middleware" file convention is deprecated. Please use "proxy" instead.`

但在当前项目里，`middleware.ts` 是 **有意保留** 的。

原因是：

- 本项目当前仍使用 `@cloudflare/next-on-pages`
- 实测直接迁移到 `proxy.ts` 会影响当前 Cloudflare 构建兼容性
- 因此当前阶段优先保证 Cloudflare Pages 自动部署稳定

所以，**不要在未迁移部署方案前，直接把 `middleware.ts` 改成 `proxy.ts`。**

当前这是一个“已知但可接受”的兼容性警告，不是线上故障。

### 5. 当前依赖和脚本层面的兼容性取舍

当前兼容性层面的关键事实：

- 项目使用 Next.js 16
- `@cloudflare/next-on-pages` 对 Next 16 并不理想，属于“可用但偏脆弱”的组合
- `.npmrc` 中启用了 `legacy-peer-deps=true`
- 构建脚本使用了 `cross-env`
- `build` / `build:cf` 中已关闭 Next 和 Vercel telemetry，以减少日志噪音

因此，以下内容不要随便删除：

- `.npmrc`
- `cross-env`
- `build:cf` 相关脚本

## 已完成的整理工作

本项目已经完成过一轮较保守的整理，主要包括：

- 删除未引用的 UI 原语文件
- 删除未引用的默认 SVG 资源
- 删除未引用的工具文件
- 删除未使用依赖
- 修正多个 `any`
- 收紧部分前后端接口类型
- 清理 lint 误扫构建产物的问题
- 保留 Cloudflare 可构建方案的前提下，减少部署日志噪音
- 去除数据库入口中的 `direct eval` bundling 警告

这轮整理的目标是：

- 瘦身
- 提高鲁棒性
- 减少死代码与维护负担
- 但不改变功能和部署链路

---

## 后续继续优化时的建议顺序

如果未来继续做“瘦身 / 精简 / 整理”，建议按下面顺序推进：

1. 先删未引用代码和资源
2. 再处理类型问题和明显重复逻辑
3. 再拆大文件
4. 最后才考虑运行时或部署链路迁移

原因是前 3 类属于低风险清理，第 4 类属于中风险或高风险改造。

---

## 允许接受的当前告警

目前以下告警可以接受，不属于阻塞问题：

- Next.js 提示 `middleware` 已废弃
- Edge Runtime 导致某些页面不能静态生成的提示
- `@cloudflare/next-on-pages` 自身已 deprecated 的提示

其中最重要的一条是：

- `middleware` 告警当前不能只凭“看起来过时”就直接改掉

---

## 真正需要谨慎处理的未来事项

如果未来要继续升级项目，优先级最高的中期事项是：

- 评估从 `@cloudflare/next-on-pages` 迁移到 OpenNext 的 Cloudflare 方案

但这不属于简单清理，应该视为一项独立迁移任务，必须单独验证：

- 本地开发
- 本地构建
- Cloudflare Pages Preview / Production
- D1 绑定
- 所有 API 路由
