# ShuttleArena - 技术方案

## 一、架构概览

```
用户（手机/PC 浏览器）
        │
        ▼
  Cloudflare CDN
        │
        ├── 静态页面 → Cloudflare Pages（Next.js SSG/SSR）
        │
        └── API 请求 → Cloudflare Workers（Next.js API Routes）
                │
                ▼
          Cloudflare D1（SQLite 数据库）
```

## 二、技术栈详情

| 层 | 技术 | 版本 | 说明 |
|----|------|------|------|
| 框架 | Next.js | 14+ (App Router) | 前后端一体 |
| UI 库 | shadcn/ui | 最新 | 基于 Radix，高质量组件 |
| 样式 | Tailwind CSS | 3.x | 原子化 CSS，响应式简单 |
| 语言 | TypeScript | 5.x | 全栈类型安全 |
| 数据库 | Cloudflare D1 | — | SQLite，免费，不暂停 |
| ORM | Drizzle ORM | 最新 | 轻量，原生 D1 支持 |
| 部署 | Cloudflare Pages | — | git push 自动部署 |
| 适配器 | @opennextjs/cloudflare | — | Next.js 运行在 CF Workers 上 |
| 认证 | 简单用户名+密码 | — | 管理员和运动员登录，极简注册 |

## 三、数据库设计（初版）

```sql
-- 用户账号（管理员 + 运动员）
users (
  id, username, password_hash,
  role [admin/athlete],
  player_id [nullable, 绑定到具体运动员代号],
  created_at
)

-- 赛事
tournaments (
  id, name, status [draft/active/finished],
  courts_count, round_duration_minutes,
  scoring_mode, scoring_points, event_date,
  start_time, end_time,
  created_at, updated_at
)

-- 参赛者（= 代号，抽签前无真名）
players (
  id, tournament_id,
  name [nullable, 抽签后填入真名],
  gender [M/F],
  group_id, position_number,
  -- 代号自动生成：如 group.icon + position_number → 🐯虎-2
  created_at
)

-- 小组
groups (
  id, tournament_id,
  name,           -- 如「虎队」
  icon,           -- 如「🐯」
  sort_order,
  created_at
)

-- 比赛模板（位置定义）
template_positions (
  id, tournament_id, position_number, gender [M/F]
  -- 如: 1-M, 2-M, 3-M, 4-F, 5-F
)

-- 比赛模板（对阵定义）
template_matches (
  id, tournament_id, match_type [MD/WD/XD],
  home_pos_1, home_pos_2, away_pos_1, away_pos_2,
  sort_order
  -- 如: MD, 1, 2, 1, 2 → 男双 1+2 vs 1+2
)

-- 比赛（实际生成的每场比赛）
matches (
  id, tournament_id,
  round_number, court_number,
  home_group_id, away_group_id,
  template_match_id,
  match_type [MD/WD/XD],
  home_player_1_id, home_player_2_id,
  away_player_1_id, away_player_2_id,
  status [pending/in_progress/finished],
  winner [home/away/null],
  created_at, updated_at
)

-- 比分（每局）
match_games (
  id, match_id, game_number,
  home_score, away_score,
  winner [home/away]
)

-- 裁判志愿记录
referee_records (
  id, match_id, player_id,
  role [referee/line_judge],
  created_at
)
```

### 代号系统设计

预设动物库（按需自动分配）：
```
🐱 猫队, 🐶 狗队, 🐰 兔队, 🐼 熊猫队, 🐹 仓鼠队,
🦝 浣熊队, 🐨 考拉队, 🦔 刺猬队, 🦦 水獭队, 🐧 企鹅队
```

代号生成规则：`{队伍icon}{队名}-{位号}` → 如 `🐱猫-1`、`🐰兔-3`

显示逻辑：
- 未绑定真人时：`🐱猫-1 (男)`
- 绑定真人后：`🐱猫-1 张三`

## 四、核心算法

### 4.1 比赛生成算法
```
输入: n个小组, 比赛模板(k场)
输出: C(n,2) × k 场比赛

for each pair (groupA, groupB) in combinations(groups, 2):
    for each template_match in template:
        生成具体比赛（根据位置→找到对应球员）
```

### 4.2 赛程编排算法（轮次分配）
```
输入: 所有比赛, 场地数, 约束条件
输出: 每场比赛的 (轮次, 场地)

目标:
  - 每轮最多 courts_count 场比赛
  - 同一轮中，同一人不出现两次（硬约束）
  - 同一人尽量不连续轮次上场（软约束）
  - 同一人尽量不连续轮次轮空（软约束）

算法: 贪心 + 回溯
  1. 将所有比赛放入待分配队列
  2. 逐轮分配:
     a. 从队列中挑选不冲突的比赛填入本轮
     b. 优先选择: 已经休息较久的人参与的比赛
     c. 避免选择: 上一轮刚打过的人参与的比赛
  3. 重复直到所有比赛分配完毕
```

### 4.3 排布质量评估
```
对每位运动员:
  - 提取其参与比赛的轮次序列
  - 计算: 总上场轮次、最大连续上场、最大连续轮空
  - 标记异常（连续≥3）
```

## 五、页面路由设计

```
公开页面（无需登录）:
  /                         → 首页（当前赛事概况）
  /schedule                 → 赛程矩阵
  /match/[id]               → 单场比赛详情 + 记分入口
  /match/[id]/scoring       → 裁判记分界面
  /standings                → 排名/积分榜
  /stats                    → 数据统计
  /stats/players            → 个人统计
  /stats/combinations       → 组合统计
  /stats/referees           → 裁判志愿榜🏅
  /history                  → 历史赛事

用户页面（需登录）:
  /login                    → 登录（管理员+运动员共用）
  /register                 → 注册（用户名+密码，极简）
  /me                       → 运动员个人中心（我的比赛、我的统计）

管理后台（管理员登录）:
  /admin/tournament         → 赛事配置（模拟器）
  /admin/players            → 人员管理 + 代号绑定
  /admin/template           → 比赛模板配置
  /admin/groups             → 分组管理
  /admin/schedule           → 赛程生成/管理
  /admin/scoring            → 批量比分录入
  /admin/users              → 用户管理（绑定运动员账号↔代号）
```
